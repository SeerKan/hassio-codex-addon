# Codex Agent Documentation

## What This Add-on Does

Codex Agent runs the official Codex CLI inside a Home Assistant OS add-on and
wraps it with Home Assistant-specific context, risk classification, backups,
audit retention, and an ingress UI.

The v1 UI supports three modes:

- **Ask**: read-only explanation and troubleshooting.
- **Propose**: read-only patch or action plan without modifying files.
- **Apply**: allows Codex to make changes after the add-on risk gate passes.

## Authentication

Each Home Assistant administrator signs in to Codex independently. The add-on
sets a separate `CODEX_HOME` for every `X-Remote-User-Id` value provided by
Home Assistant ingress.

The preferred login path is device-code login from the sidebar. If that is not
available in your OpenAI workspace, use the fallback import flow and paste an
`auth.json` generated by `codex login` on a trusted machine.

`auth.json` contains sensitive access and refresh tokens. Do not share it, paste
it into issue trackers, or commit it to git.

## Home Assistant Access

The add-on uses:

- Home Assistant ingress for the sidebar UI.
- `panel_admin: true` so only administrators see the panel.
- `homeassistant_api: true` for the Core REST and WebSocket API proxy.
- `hassio_api: true` with `hassio_role: admin` for Supervisor APIs.
- Mapped Home Assistant folders so Codex can inspect and change configuration.

The add-on does not publish a host port. The web UI is intended to be reached
through Home Assistant ingress only.

## Backups

When `create_backup_before_first_change` is enabled, the add-on creates a full
Supervisor backup before the first configuration-changing apply-mode or
full-auto run. Runtime entity-control requests do not create a backup. The
backup is created once for the add-on installation and the slug is stored in
`/data`.

If the backup fails, the run is blocked.

## Risk Gates

Low-risk requests can run automatically. High-risk requests require explicit
approval before a Codex run starts.

High-risk examples:

- delete, purge, wipe, factory reset, or destructive restore
- direct access to `secrets.yaml`, private keys, tokens, or credential files
- replacing authentication, users, MFA, or network access controls
- Supervisor restore, host shutdown, or broad add-on operations

Full-auto mode is available only when `allow_yolo_mode` is enabled. It runs
Codex with `--dangerously-bypass-approvals-and-sandbox`, and should only be used
when you are prepared to restore from backup.

## Retention

The add-on stores prompts, risk assessments, Codex events, final messages, and
generated diffs for `retention_days`, defaulting to 30. Per-user Codex auth
state is not removed by retention cleanup.

## Publishing

For public distribution, publish multi-architecture images to GHCR and keep the
generic image name in `config.yaml`:

```yaml
image: ghcr.io/seerkan/hassio-codex-addon
```

If you publish under a different GitHub owner or repository, update:

- `repository.yaml`
- `codex_agent/config.yaml`
- `.github/workflows/release.yaml`
- README installation links

## Current v1 Boundaries

- HA OS is the target install type.
- Sidebar UI only; Assist/voice integration is intentionally out of scope.
- Secret-file access requires explicit approval.
- The add-on relies on the Codex CLI contract and may need updates when Codex
  CLI authentication output changes.
